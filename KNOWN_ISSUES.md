# Known Issues

Running punch-list of confirmed bugs and open questions in OrcStompTiming, to tackle deliberately rather than all at once. This file lives in git on purpose — it's the source of truth across machines and sessions, not anyone's local notes.

Context: this is a DIY RFID 5k timing rig built around a cheap Chinese UHF reader (manufacturer `aosid.com`, type code `ABQ`, firmware `V1.22`), built as a proof of concept to avoid a $5k commercial solution. The codebase has rough edges by design — this list exists so fixes get applied one at a time, deliberately, instead of guessed at from scratch each session. See [PROTOCOL_SPEC.md](PROTOCOL_SPEC.md) for the reader's command protocol and [PROJECT_GUIDELINES.md](PROJECT_GUIDELINES.md) for the rules governing how changes get made here.

## Open Issues

1. **Mode-switch command is fabricated, not real protocol — proven false, not just unverified.** `AppUI.setReaderMode()` (`AppUI.js`) sends `7CFFFF010201XX` (CID1=01H, CID2=02H). The protocol's CID2 code table (`PROTOCOL_SPEC.md` Section 3, Table 3-2) is a *closed* list of exactly four action codes (`31H`/`32H`/`21H`/`22H`) — `02H` isn't a documentation gap, it's provably not a real command. The real control is Section 4.8/4.9 "Set/Get Basic Parameter of Reader" (CID1=81H, CID2=31H to set / 32H to get), a 28-byte payload with a **Work Mode (WM)** byte buried inside: `0x01`=Command (quiet/on-demand), `0x02`=Active (auto-broadcast), `0x03`=Passive. Correct fix: **Get** the current 28-byte block → flip only the WM byte → **Set** the same block back, so RF power/frequency aren't clobbered. Fully documented in `PROTOCOL_SPEC.md` Section 4.8; not yet implemented in code.

2. **Read TID (Bank 2) is likely blocked by #1, not actually broken itself.** The command `7CFFFF123203020006` (`index.html`) is protocol-correct — CID1=12H, CID2=32H, MB=02 (TID), SA=00, DL=06 words, matches `PROTOCOL_SPEC.md` Section 4.7 exactly. Theory: since mode-switch never worked, the reader is still in factory Active mode, constantly auto-broadcasting reads, and on-demand commands collide with that stream. The `0x1F` "error" referenced in earlier notes does **not** appear anywhere in the manual (only `00`=Success, `01`=Fail, `32`=Auto-send-to-SU are real RTN codes per Table 2-3) — unlike #1, this one is genuinely unverified, not disproven; the manual not mentioning it doesn't mean this clone reader doesn't really emit it. Tracked as "Set Aside" item 2 in `PROTOCOL_SPEC.md` Section 6. Fix #1 first; this may resolve itself.

3. **Suspected off-by-one in EPC extraction, and the whole Active-mode push-frame shape it depends on is undocumented.** `BleDriver.js`, `processValidFrame()`:
   ```js
   const epcLen = frame[6];
   const tagBytes = frame.slice(6, 6 + epcLen);
   ```
   `frame[6]` is meant to be the EPC_LEN byte itself, with EPC bytes following — but `CID1=0x20` does not exist anywhere in the manual's command table, and there's no documented frame layout for an unsolicited Active-mode push at all (the manual only says RTN comes back `0x32`, never shows the frame). So this isn't just a one-byte slip in an otherwise-confirmed layout — the entire premise (`frame[3]===0x20`, EPC_LEN at `frame[6]`) is empirically reverse-engineered from real device traffic with zero documentation backing it. `frame.slice(7, 7+epcLen)` may be the right fix if EPC_LEN really is at index 6, but that premise itself needs confirming first — by sniffing several real Active-mode push frames against known tags. See `PROTOCOL_SPEC.md` Section 6, item 1.

4. **`MockBleDriver` frame shape disagrees with the real `BleDriver`.** `MockBleDriver.js` strips tag hex with fixed offsets (`substring(6, len-4)`), assuming no EPC_LEN byte exists — but the real parser explicitly reads one at `frame[6]`. Mock-mode "Simulate 10s Window" testing therefore doesn't exercise the real parsing path faithfully. Low priority since Mock mode is a dev convenience, but it can mask real-parser bugs.

5. **Ingestion-side dedup is documented but not implemented.** `PROJECT_GUIDELINES.md` §4 calls for an "Ingestion Dedup" (don't save the same chip 100x/sec) ahead of the Peak-RSSI export-time dedup. `TimingEngine.handleIncomingTag()` pushes every single read straight to the write queue with no filtering at all. Only the export-time Peak-RSSI 10s-window logic exists (and that one is correct and tested). In real Active-mode scanning against an antenna this could make `race_reads` grow very large during a live race. Not urgent for a one-shot test but will matter at race scale.

6. **Read vs. Write `SA` pointer units disagree within the manual itself.** Section 4.6 says Write's `SA` is a byte pointer; Section 4.7 says Read's `SA` is a word pointer — both use the same example value. This directly affects whether `writeBibToTag()`'s assumption about EPC word offset 2 is correct (see Unconfirmed list below). Documented in `PROTOCOL_SPEC.md` Section 5.

7. **`establishConnection()` reports "READER ONLINE" success even if zero characteristics were ever found.** `BleDriver.js` never checks that `notifyCharacteristic`/`writeCharacteristic` are non-null before declaring success — if no service matches the `ffe` filter at all, both stay `null`, `startNotifications` never gets called, and the UI still shows connected. Pinned down by a passing characterization test in `tests/connection.test.js` ("(known gap) currently reports READER ONLINE even when no service matches..."), which documents today's actual behavior, not desired behavior. Per the Testing Discipline rule in `PROJECT_GUIDELINES.md` §7, fixing this means flipping that test's assertion to `rejects` first (watch it fail), then fixing the code.

8. **`parseFrame()` has no cross-notification reassembly buffer.** `BleDriver.js` parses each `characteristicvaluechanged` event's bytes in isolation with zero carry-over. If a single protocol frame's total length ever exceeds the negotiated BLE notification payload size and arrives split across two or more separate events, the first partial chunk fails its `frameEnd > bytes.length` check and is silently dropped — no warning, nothing. Not currently confirmed to be biting anything (a real 41-byte Get Version reply decoded fine, implying this hardware/Chrome/Windows combo negotiates a large-enough MTU in practice), but a longer response (e.g. multi-tag identify with several EPCs) could hit it. Would need a deliberately oversized real response to confirm the actual risk either way.

## Resolved

- **Stale regression test rewritten** (`tests/regression.test.js`, "concatenated tag frames" test). The old version used a fixed 7-byte mock frame with no real LENGTH field — it modeled a format that was never real, predating both the protocol read-through and the CID1-based dispatch logic. Rewritten with protocol-accurate frames (real LENGTH-based framing, verified checksum) that faithfully characterize current parser behavior, including the known EPC off-by-one (#3 above) rather than silently "fixing" it via the test.
- **"Bit Flip Test" fixed to fail for the correct reason.** The old version's mock frame got rejected by the *framing* check before checksum validation was ever reached, so it passed without exercising checksum-mismatch logic at all. Rebuilt with correct framing and a deliberately bit-flipped CHKSUM byte, plus a `console.warn` spy proving the rejection now genuinely happens in the checksum branch.
- **"Connect Reader" flakiness investigated and mitigated.** Symptom: `Discovery Error: Connection Error: Connection attempt failed.` on every attempt. Two distinct failure modes share that symptom:
  1. *Ordinary transient flakiness* — `device.gatt.connect()` is known-flaky on Windows right after a BLE scan. Fixed: `BleDriver.js`'s `connectGatt(maxAttempts=4)` retries with backoff and an explicit `gatt.disconnect()` between attempts before giving up.
  2. *Wedged OS Bluetooth radio* — confirmed via `chrome://bluetooth-internals/#devices` showing the device as "Not Connected" (ruling out a stuck Chrome GATT session or another tab holding it). No code can fix this — Web Bluetooth has no permission to reset the OS radio. The only real fix was toggling Bluetooth off/on in Windows Settings.
  - Also fixed: `connect()` and `establishConnection()` used to both call `updateStatus()` on the same bubbled-up error, producing a confusing nested `"Discovery Error: Connection Error: ..."` message. Now cleanly separated — picker-level failures say "Discovery Error", GATT-level failures say "Connection Error", each exactly once.
  - **Runbook for next time this recurs:** check `chrome://bluetooth-internals/#devices` first. "Connected" → something (another tab, a stale handle) is holding it, force-disconnect there. "Not Connected" but still failing after retries → it's the OS radio, toggle Bluetooth off/on in Windows Settings and/or power-cycle the reader.

## Unconfirmed / Low Priority

- `BleDriver.processValidFrame`'s RSSI sign handling (`rssiRaw > 127 ? rssiRaw - 256 : -rssiRaw`) looks like a reverse-engineered heuristic rather than something pulled from a documented RSSI field — the manual never mentions RSSI at all. Worth a second look if RSSI-based decisions (e.g. Peak-RSSI export logic) ever look suspect.
- `AppUI.writeBibToTag()` assumes the bib number is written as a raw 16-bit value directly into the EPC bank at word offset 2 (skipping the PC word) — unverified against spec or a real written/read-back tag. Complicated by #6 above: if `SA` for EPC writes is actually a byte pointer rather than a word pointer, this write may be landing 2 bytes off from where it's assumed to be.
- The "Bit-Search Decoder" feature (`app.runBitSearch`) is a manual debug heuristic (search for a bib's binary pattern inside the raw EPC) — fine as a debug tool, just not a real decode of a structured tag ID.

## How to Use This Doc

Read this alongside `PROTOCOL_SPEC.md` and `PROJECT_GUIDELINES.md` before making changes here, regardless of which machine you're on. Pull from this list rather than re-deriving findings from scratch. Update it (move items to Resolved, add new findings) as work progresses — don't let it go stale. Per `PROJECT_GUIDELINES.md` §5 and §7: no code change without the User's explicit go-ahead, and no code change without a failing test first.
