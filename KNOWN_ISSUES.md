# Known Issues

Running punch-list of confirmed bugs and open questions in OrcStompTiming, to tackle deliberately rather than all at once. This file lives in git on purpose — it's the source of truth across machines and sessions, not anyone's local notes.

Context: this is a DIY RFID 5k timing rig built around a cheap Chinese UHF reader (manufacturer `aosid.com`, type code `ABQ`, firmware `V1.22`), built as a proof of concept to avoid a $5k commercial solution. The codebase has rough edges by design — this list exists so fixes get applied one at a time, deliberately, instead of guessed at from scratch each session. See [PROTOCOL_SPEC.md](PROTOCOL_SPEC.md) for the reader's command protocol and [PROJECT_GUIDELINES.md](PROJECT_GUIDELINES.md) for the rules governing how changes get made here.

## Open Issues

1. ~~Mode-switch command was fabricated, not real protocol.~~ **Resolved 2026-07-07** — see Resolved section below.

10. ~~**Get Parameters response is 27 bytes on this hardware, not 28 as the spec documents.**~~ **Resolved 2026-07-12** — `getReaderParameters()` and `setReaderParameters()` deleted as part of cleanup; CtrlAutoRead approach needs no parameter block access. Full protocol detail preserved in `PROTOCOL_SPEC.md` §4.8 and the Resolved entry below.

2. **TID read (Bank 2) is unverified.** Command `7CFFFF123203020006` is protocol-correct but has never been tested in Command mode. Theory: Active-mode auto-broadcasts collide with on-demand commands, so it may only work after `setWorkMode('command')`. The UI button was removed in the 2026-07-12 cleanup — if TID read is ever needed, add it back only after confirming it works in Command mode.

3. ~~`MockBleDriver` frame shape disagrees with the real `BleDriver`.~~ **Resolved 2026-06-30** — see Resolved section below.

4. ~~Ingestion-side dedup is documented but not implemented.~~ **Resolved 2026-07-01** — see Resolved section below.

5. **Read vs. Write `SA` pointer units disagree within the manual itself.** Section 4.6 says Write's `SA` is a byte pointer; Section 4.7 says Read's `SA` is a word pointer — both use the same example value. This directly affects whether `writeBibToTag()`'s assumption about EPC word offset 2 is correct (see Unconfirmed list below). Documented in `PROTOCOL_SPEC.md` Section 5.

6. ~~`establishConnection()` reports "READER ONLINE" success even if zero characteristics were ever found.~~ **Resolved 2026-06-30** — see Resolved section below.

7. **`parseFrame()` has no cross-notification reassembly buffer.** `BleDriver.js` parses each `characteristicvaluechanged` event's bytes in isolation with zero carry-over. If a single protocol frame's total length ever exceeds the negotiated BLE notification payload size and arrives split across two or more separate events, the first partial chunk fails its `frameEnd > bytes.length` check and is silently dropped — no warning, nothing. Not currently confirmed to be biting anything (a real 41-byte Get Version reply decoded fine, implying this hardware/Chrome/Windows combo negotiates a large-enough MTU in practice), but a longer response (e.g. multi-tag identify with several EPCs) could hit it. Would need a deliberately oversized real response to confirm the actual risk either way.

8. ~~Chip-to-Bib mapping has zero duplicate protection — two distinct gaps.~~ **Resolved 2026-07-05** for the kiosk form path — see Resolved below.
9. ~~Bulk CSV chip-map import bypasses the reassignment guardrails added for the kiosk form.~~ **Resolved 2026-07-05** — see Resolved below.

## Resolved

- **Tag Inspector now decodes tag-read frames into separate, labeled fields instead of just the generic header/addr/cid1/cid2/length/info/checksum grid.** `BleDriver.processValidFrame()`'s tag-decode logic (EPC_LEN, EPC bytes, RSSI raw/dBm) was extracted into a standalone `decodeTagFrame(frame)` method, reused by both the real `onTagRead` path and the new "TAG READ DECODE" panel in the Tag Inspector tab — one source of truth, so the inspector can never silently disagree with what real race scoring does. `onRawFrame` now passes `{hex, frame, checksumValid, tagDecode}` instead of a bare hex string, so the inspector can also show a ✓/✗ checksum-valid indicator it never could before (`processValidFrame` previously fired `onRawFrame` *before* even computing the checksum). Covered by 4 new tests in `tests/regression.test.js` (31/31 passing). Found and fixed in the same pass: `MockBleDriver.playSequence()` was still emitting the old bare-hex-string `onRawFrame` shape, which would have crashed the inspector in Mock Mode — updated to match the new contract (`tagDecode: null`, since mock frames don't share the real CID1=0x20 push-frame shape — see #3 below). Also found and fixed: `runBitSearch()` (`index.html`) was reading the EPC via a fragile `document.querySelector('#inspectorDetail div div:nth-child(2) span')` that actually grabbed the **Header** field's span, not the EPC — the Bit-Search Decoder had likely never searched real EPC bytes. Now reads `#inspectorEpcHex`, a stable ID on the new EPC field. Verified by rendering `AppUI.updateInspector()` against a real DOM (happy-dom) for a valid tag frame, a non-tag frame, and a bad-checksum tag frame, and inspecting the actual HTML output — no in-browser check was done (no browser-automation tool available this session).
- **EPC decode fixed and confirmed against 5 real scans (2026-06-28).** The previous "EPC_LEN byte at `frame[6]`" assumption was empirically falsified: Tony provided 4 real race tag-read frames (Bib 21251 & 5251 from one race/provider, Bib 982 & 868 from two other races/providers) plus the manufacturer's own sample/test tag bundled with the reader, and on every single one `frame[6]` was `0x00` — meaning the old code decoded an **empty-string EPC for every real tag read**, with zero ability to distinguish bibs from each other on actual hardware. All 5 checksums were hand-verified (sum of all bytes mod 256 = 0) before trusting the data. The real layout is `INFO = [AN(1B antenna)][PC(2B, standard EPC Gen2 Protocol Control word)][EPC(PC-word-derived length)][RSSI(1B)]` — EPC length is the top 5 bits of PC's high byte, in 16-bit-word units, exactly like a real Gen2 PC word. This was confirmed multiple ways: it correctly predicted a 12-byte EPC for the Race-1 pair (`PC=0x3400`) and the sample tag (`PC=0x3000`), and a 16-byte EPC for the two other-race tags (`PC=0x4000`), every time exactly accounting for the remaining `LENGTH` bytes with nothing left over — and the sample tag's decoded EPC happens to lead with `E2`, the standard Gen2 General Identifier header, which a wrong byte offset wouldn't coincidentally produce. `BleDriver.decodeTagFrame()` now implements this layout; `AppUI.updateInspector()`'s "TAG READ DECODE" panel now shows AN/PC/EPC-length/EPC/RSSI as separate fields. All 5 real frames are now permanent regression test data in `tests/regression.test.js` (36/36 passing), alongside two corrected hand-built synthetic frames (used by the concatenated-frames and bit-flip tests) that now follow the same confirmed shape instead of the old falsified one.
- **Stale regression test rewritten** (`tests/regression.test.js`, "concatenated tag frames" test). The old version used a fixed 7-byte mock frame with no real LENGTH field — it modeled a format that was never real, predating both the protocol read-through and the CID1-based dispatch logic. Rewritten with protocol-accurate frames (real LENGTH-based framing, verified checksum); updated again 2026-06-28 to use the confirmed AN+PC+EPC+RSSI shape instead of the now-falsified EPC_LEN-byte shape.
- **"Bit Flip Test" fixed to fail for the correct reason.** The old version's mock frame got rejected by the *framing* check before checksum validation was ever reached, so it passed without exercising checksum-mismatch logic at all. Rebuilt with correct framing and a deliberately bit-flipped CHKSUM byte, plus a `console.warn` spy proving the rejection now genuinely happens in the checksum branch.
- **"Connect Reader" flakiness investigated and mitigated.** Symptom: `Discovery Error: Connection Error: Connection attempt failed.` on every attempt. Two distinct failure modes share that symptom:
  1. *Ordinary transient flakiness* — `device.gatt.connect()` is known-flaky on Windows right after a BLE scan. Fixed: `BleDriver.js`'s `connectGatt(maxAttempts=4)` retries with backoff and an explicit `gatt.disconnect()` between attempts before giving up.
  2. *Wedged OS Bluetooth radio* — confirmed via `chrome://bluetooth-internals/#devices` showing the device as "Not Connected" (ruling out a stuck Chrome GATT session or another tab holding it). No code can fix this — Web Bluetooth has no permission to reset the OS radio. The only real fix was toggling Bluetooth off/on in Windows Settings.
  - Also fixed: `connect()` and `establishConnection()` used to both call `updateStatus()` on the same bubbled-up error, producing a confusing nested `"Discovery Error: Connection Error: ..."` message. Now cleanly separated — picker-level failures say "Discovery Error", GATT-level failures say "Connection Error", each exactly once.
  - **Runbook for next time this recurs:** check `chrome://bluetooth-internals/#devices` first. "Connected" → something (another tab, a stale handle) is holding it, force-disconnect there. "Not Connected" but still failing after retries → it's the OS radio, toggle Bluetooth off/on in Windows Settings and/or power-cycle the reader.

- **Connection hardening + persistent pairing (2026-06-30).** Five things landed together: (1) `establishConnection()` now throws "No usable characteristics found" instead of silently reporting READER ONLINE when no `ffe` service is present — the old characterization test was flipped to `rejects` first per §7. (2) `requestDevice()` uses `acceptAllDevices: true` with `optionalServices: ['ffe0', 'ffe1']` — a `filters: [{services: ['ffe0']}]` approach was attempted but reverted (2026-06-30) because this hardware does not include `ffe0` in its BLE advertisement packets, causing the device picker to show nothing. (3) `connect()` saves `device.id` / `device.name` to `localStorage` after a successful pairing. (4) `disconnect()` always clears those keys — intentional disconnect = forget the device, so auto-reconnect can't silently re-pair to a "forgotten" reader. (5) New `tryAutoConnect()` method uses `navigator.bluetooth.getDevices()` (Chrome 85+) to reconnect without showing the picker if the previously paired device is available; `AppUI.restoreState()` calls it on page load. **Confirmed non-functional on Windows Chrome (2026-06-30):** `getDevices()` consistently returns an empty list even immediately after a successful `requestDevice()` grant on `localhost` — the code is correct but the platform doesn't honour it. No JS workaround exists; see WISHLIST Auto-Connect entry. UI: "Connect Reader" button becomes "Disconnect" when connected (label managed by `updateBleBadge`); a "Forget Device" button appears when disconnected with a saved device, to manually clear the saved pairing without having to connect first. Covered by 11 new tests in `connection.test.js` and 5 new tests in `inspector-ui.test.js`.

- **L2 ingestion dedup implemented (2026-07-01).** `TimingEngine` now maintains a `seenTags: Map<tagHex, firstSeenMs>` that gates writes in `startDaemon()`. The first read for a tag always writes; subsequent reads within the 10-second window also write (so L3 peak-RSSI selection still works correctly across the full window); reads arriving after the window closes are silently dropped at the daemon, not written to IndexedDB at all. `clearRaceData()` clears the map so each new race starts fresh. This resolves the DB-growth concern without touching L3 (export-time peak selection is unchanged and still correct). Covered by 7 tests in `tests/dedup.test.js`.

- **`MockBleDriver` frame shape fixed (2026-06-30).** `MockBleDriver.js` now builds protocol-accurate `CID1=0x20` push frames (`AN+PC+EPC+RSSI` layout) via `buildTagFrame(epcHex, rssiDbm)`, using the same PC-word encoding that real frames use. `playSequence()` and `startSimulation()` both call `buildTagFrame()` → `decodeTagFrame()` (an inline mirror of `BleDriver.decodeTagFrame()`) and emit the result as a proper `tagDecode` object in `onRawFrame` — so Mock Mode now exercises the same parse path as real hardware, and the Tag Inspector shows live decode output in Mock Mode. The old `tag`-as-raw-frame-hex sequence format has been replaced with `{ epcHex, rssi, delay }`. Covered by `tests/mock-driver.test.js` (19 tests), including a cross-validation test that independently runs `BleDriver.decodeTagFrame()` against the frames `MockBleDriver` emits to confirm they agree.

- **Tag Inspector's WIP/dangerous controls are now disabled in the UI (2026-06-28), not just documented as risky.** The Pro Tag Writer ("Burn to Tag"), Pro Diagnostics (mode-switch, Get Version, Read TID), Manual Command (Advanced) (both send buttons), and Bit-Search Decoder controls are all `disabled` in `index.html` — kept visible with a "(WIP — disabled, see KNOWN_ISSUES.md)" label rather than removed, so the half-finished/unverified features (#1, #2, and the two Unconfirmed items above) can't be triggered by accident (e.g. mid-race) before they're actually fixed. The passive Tag Inspector display (Last Scanned Frame, Frame History) is untouched — only the active/write/command controls are gated. Covered by a new `tests/inspector-ui.test.js` (5 tests, loads the real `index.html` body into happy-dom and asserts each control's `disabled` state), so a future edit can't silently re-enable one of these without a test failing.

- **Chip-to-Bib reassignment guardrails added for the kiosk form (2026-07-05).** Closes both gaps in old item #8 for manual/kiosk entry. `TimingEngine.saveMapping()` now coerces `bib_num` via `parseInt()` regardless of caller type (fixes a related bug found this session: the kiosk form passed a raw `<input>.value` string while CSV import already `parseInt`'d, so the same bib could be stored as `"104"` and `104` and never compare equal). New `AppUI.checkMappingConflicts(chipHex, bibNum, maps)` detects both a chip-side conflict (this chip already mapped to a different bib) and a bib-side conflict (this bib already held by a different chip); `AppUI.submitMapping()` shows a `confirm()` naming the conflicting side(s) before saving, and does nothing silently — cancelling leaves the mapping untouched. `AppUI.refreshMappingFormState()` reflects this live as the operator types/scans: a high-vis "ALREADY MAPPED" banner (`#mappingConflictWarning`) and the submit button relabeling to "REASSIGN" (`#mappingSubmitBtn`), wired via `oninput` on both fields and called at the end of `fillKioskForm()` so a kiosk scan of an already-registered chip shows the warning immediately, before the operator types anything. Also fixed while in the area: `renderMappingTable()` now escapes `chip_hex`/`bib_num` before interpolating into `innerHTML` (a crafted CSV import row could previously inject markup into the registry table). Caught via manual browser smoke-testing (Playwright + headless Chromium), not just unit tests: the first implementation warned+confirmed a bib-side steal but only wrote the new chip's row, leaving both chips sharing the bib — `chip_map` is keyed by `chip_hex`, so `saveMapping()` had no way to touch the losing chip's row. Fixed by adding `TimingEngine.deleteMapping(chipHex)` and having `submitMapping()` call it on the losing chip once a bib steal is confirmed; `index.html`'s `app.deleteMapping` now also delegates to it instead of touching IndexedDB directly. Covered by `tests/chip-map.test.js` (38 tests).

- **Bulk CSV chip-map import rebuilt as an all-or-nothing full replace (2026-07-05), per Tony's direction.** Old `app.importChipMap` looped calling `saveMapping()` per row with zero validation — same duplicate-bib/silent-overwrite risk as #8, but for an entire file at once. New `AppUI.parseChipMapCsv(text)` parses rows (normalized like `saveMapping`, tagged with the 1-indexed source line number) and `AppUI.findCsvConflicts(rows)` checks the file's own internal consistency *before* it ever touches `chip_map`: a chip_hex repeated across rows, or a bib_num claimed by two different chips, both refuse the import via `alert()` naming the exact conflicting chip/bib and line numbers — no partial import, existing registry left untouched. A file with zero valid rows is also refused (guards against silently wiping the registry on a garbage/empty upload). If the file is clean, `AppUI.submitChipMapImport()` shows a `confirm()` making clear this **replaces the entire registry** (not a merge — anything not in the file is deleted), and only calls new `TimingEngine.replaceChipMap(mappings)` (clears `chip_map`, inserts the file's rows in one transaction) if accepted. `index.html`'s `app.importChipMap` is now a thin wrapper. Verified in a real browser (Playwright + headless Chromium): conflicting file refused with correct line numbers and registry untouched, clean file's overwrite warning shown, cancel preserved the existing registry, accept replaced it — zero console errors. Covered by `tests/chip-map.test.js` (54 tests total in that file).

- **Mode switch fully resolved (2026-07-07), confirmed on hardware.** Active mode now restores correctly. Root cause: `SET Parameters WM=0x02 + software reset` only updates the flash-stored work mode — it does not restart the runtime scan loop. Fix: replaced `setWorkMode()` in `BleDriver.js` with `CtrlAutoRead` (CID1=0x34, undocumented MM-reader extension found in `ADDeviceReader.dll` via ildasm). `setWorkMode('command')` → `sendRawHex('7CFFFF34000100')` (status=0, stops scan loop); `setWorkMode('active')` → `sendRawHex('7CFFFF34000101')` (status=1, starts scan loop). Both return `CCFFFF34000002` (RTN=00 success) and work correctly. `StopAutoRead` (CID1=0x33) was tried first — not recognized on this hardware (zero response). See `PROTOCOL_SPEC.md` §6 for full protocol details. Covered by `tests/mode-switch.test.js` (10 tests).

- **`getReaderParameters()` / `setReaderParameters()` deleted (2026-07-12).** The 27-vs-28-byte quirk (#10) was the main reason these methods existed. Since `setWorkMode()` now uses CtrlAutoRead exclusively and never touches the parameter block, these methods had no production caller and were removed. The 27-byte-Get / 28-byte-Set asymmetry and the MR padding requirement are documented in `PROTOCOL_SPEC.md` §4.8 for future reference.

- **Mode switch partial fix (2026-07-05 / 2026-07-06); Active mode still broken — REOPENED as #1.** `AppUI.setReaderMode()` (fabricated CID1=01H/CID2=02H) and `AppUI.writeBibToTag()` (unverified placeholder) were deleted. `BleDriver` now has `sendCommand(hexString, expectedCid1, timeoutMs)` (Promise-based command/response with CID1 matching, Active-mode push-frame passthrough, and timeout), `getReaderParameters()` (CID1=81H/CID2=32H → 27-byte INFO block on this hardware), `setReaderParameters(paramBytes)` (CID1=81H/CID2=31H → always sends 28 bytes LEN=0x1C, padding if needed — see #10 for why), and `setWorkMode(mode)` (gets params → flips WM byte at index 9 → sets back → software reset for Active mode). Command mode works. Active mode does NOT return the reader to emitting push frames. Root cause: `SET Parameters` doesn't restart the scan loop; a dedicated `CtrlAutoRead(1)` command (CID1=0x34, undocumented MM extension) is needed — see `PROTOCOL_SPEC.md` §6. Fix is planned but not yet implemented. Covered by `tests/ble-command.test.js` (6 tests) and `tests/mode-switch.test.js` (9 tests).

- **Bib write command corrected to CID1=0x22 + poll-first singulation (2026-07-07/08).** Two bugs found and fixed. (1) The documented CID1=0x12 write produces no response — the real command is CID1=0x22 (undocumented MM extension, found in USB traffic capture 2026-07-07). (2) Sending CID1=0x22 without first singulating the chip causes a false RTN=0x00 — the reader accepts the frame but writes nothing to the chip (confirmed 2026-07-08 via hardware: write returned "success" but chip still read FFFFFFFF). Fix: `writeBibToEpc()` now polls with CID1=0x20 first (in command mode) to singulate the tag, then immediately sends CID1=0x22 to the held-singulated tag. Also changed DL from 02 to 06 to write the full 96-bit EPC (matching the vendor demo app — DL=02 partial writes also produced false RTN=0x00 on this hardware). Current frame: `7C FF FF 22 00 13 00000000 01 02 06 4F53BHBL 0000000000000000 [CHKSUM]` (trailing zeros, not FFs — zeros read as intentionally empty; FFs look like unwritten factory data). See `PROTOCOL_SPEC.md` §6.0 and §6.1. Covered by `tests/bib-write.test.js` (10 tests).

- **Bib Programming mode added (2026-07-05, revised 2026-07-08).** Replaces the old Pro Tag Writer WIP placeholder. `BleDriver.writeBibToEpc(bibNum)` writes the magic prefix `0x4F53` + 16-bit bib big-endian at word 2 of EPC bank, filling the remaining 8 EPC bytes with zeros. `AppUI.decodeBibFromEpc(epcHex)` recognizes the `4F53` prefix in any incoming tag EPC; `buildResultsFromReads()` uses the EPC-decoded bib before falling back to `chip_map`. The chip_map workflow is unchanged — still a fallback for chips not programmed with this app. The "Bib Programming" section in the Tag Inspector tab: "Start Programming Session" pauses the scan loop via `CtrlAutoRead(0)`; "Write to Chip" polls for the chip with CID1=0x20 (reader stays in command mode, up to 5s), writes via CID1=0x22 once singulated; "End Programming Session" resumes the loop via `CtrlAutoRead(1)`. Covered by `tests/bib-write.test.js` (10 tests) and 3 EPC-path tests in `tests/export.test.js`.

- **Race-day backup system added (2026-07-05)** — periodic lightweight live snapshots, a heavyweight backup before any destructive clear, and stronger wipe-confirm phrasing. Full detail logged in `WISHLIST.md` under "Race-Day Backup System" rather than duplicated here; see also `PROJECT_GUIDELINES.md` §8 for the human-backup redundancy plan (camera/track timer/barcode scanner) this complements. One real bug caught along the way, worth calling out: the first version of `AppUI.chooseBackupLocation()` had no error handling at all — a user cancelling the native file picker (a completely normal `AbortError`) or any `saveBackupHandle()` failure would silently break with zero feedback. Fixed with a try/catch: cancelling is now a quiet no-op, any other failure surfaces an `alert()`. Covered by `tests/backup.test.js` (21 tests).

## Future / Wishlist

- **Start line reader — true chip time and DNS/DNF separation.** Currently there is no start reader, so chip time = gun time for all finishers. A second reader at the start line would record each athlete's individual start crossing, enabling true chip time (finish read − start read). It would also distinguish DNS (no start read recorded) from DNF (start read exists but no finish read). Architecture impact: `TimingEngine` would need a second read store or a `type` field on reads (`'start'` vs `'finish'`), and `buildResultsFromReads` would need to join them. No urgency until a start reader is available.

- **EPC Permalock button — permanent write-protection after programming.** EPC Gen2 chips support a Permalock command that permanently prevents any future writes to the EPC bank (no password, no override, irreversible). Adding a "Confirm & Lock" button next to "Scan to Verify" would let Tony permalock each chip after verifying the bib is correct — eliminating any risk of accidental or malicious reprogramming. Current access password is `0x00` (unprotected); permalocking requires no password change and is the cleanest option for single-use race chips. Downside: a wrongly-programmed chip that gets permalocked is scrap, so it must come after verification. Requires a new `BleDriver.permalockEpc()` method using the EPC Gen2 Lock command.

- **EPC encoding may need to expand beyond 4 bytes someday.** Currently the OrcStomp encoding is `4F53` (magic prefix "OS") + 2-byte bib big-endian + 8 zero bytes — we use 4 of the 12 available EPC bytes and leave the other 8 as zeros. If we ever want to encode additional data (wave number, gender, age group, etc.) those 8 bytes are available. The zeros make it clear they're intentionally unused. The only constraint is that `decodeBibFromEpc` and `writeBibToEpc` stay in sync with each other — encoding and decoding are currently both hardcoded to the same 4-byte scheme. **Observed competitor encoding (race 2026-07-05):** chips read as `00 00 00 00 00 00 00 00 00 02 84` for bib 284 — no magic prefix, bib at the end of the EPC in packed BCD (`02 84` = packed BCD for 284). No inter-op required; noted for context.

## Unconfirmed / Low Priority

- `BleDriver.processValidFrame`'s RSSI sign handling (`rssiRaw > 127 ? rssiRaw - 256 : -rssiRaw`) looks like a reverse-engineered heuristic rather than something pulled from a documented RSSI field — the manual never mentions RSSI at all. Worth a second look if RSSI-based decisions (e.g. Peak-RSSI export logic) ever look suspect.

## How to Use This Doc

Read this alongside `PROTOCOL_SPEC.md` and `PROJECT_GUIDELINES.md` before making changes here, regardless of which machine you're on. Pull from this list rather than re-deriving findings from scratch. Update it (move items to Resolved, add new findings) as work progresses — don't let it go stale. Per `PROJECT_GUIDELINES.md` §5 and §7: no code change without the User's explicit go-ahead, and no code change without a failing test first.
