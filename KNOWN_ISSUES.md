# Known Issues

Running punch-list of confirmed bugs and open questions in OrcStompTiming, to tackle deliberately rather than all at once. This file lives in git on purpose — it's the source of truth across machines and sessions, not anyone's local notes.

Context: this is a DIY RFID 5k timing rig built around a cheap Chinese UHF reader (manufacturer `aosid.com`, type code `ABQ`, firmware `V1.22`), built as a proof of concept to avoid a $5k commercial solution. The codebase has rough edges by design — this list exists so fixes get applied one at a time, deliberately, instead of guessed at from scratch each session. See [PROTOCOL_SPEC.md](PROTOCOL_SPEC.md) for the reader's command protocol and [PROJECT_GUIDELINES.md](PROJECT_GUIDELINES.md) for the rules governing how changes get made here.

## Open Issues

1. **Mode-switch command is fabricated, not real protocol — proven false, not just unverified.** `AppUI.setReaderMode()` (`AppUI.js`) sends `7CFFFF010201XX` (CID1=01H, CID2=02H). The protocol's CID2 code table (`PROTOCOL_SPEC.md` Section 3, Table 3-2) is a *closed* list of exactly four action codes (`31H`/`32H`/`21H`/`22H`) — `02H` isn't a documentation gap, it's provably not a real command. The real control is Section 4.8/4.9 "Set/Get Basic Parameter of Reader" (CID1=81H, CID2=31H to set / 32H to get), a 28-byte payload with a **Work Mode (WM)** byte buried inside: `0x01`=Command (quiet/on-demand), `0x02`=Active (auto-broadcast), `0x03`=Passive. Correct fix: **Get** the current 28-byte block → flip only the WM byte → **Set** the same block back, so RF power/frequency aren't clobbered. Fully documented in `PROTOCOL_SPEC.md` Section 4.8; not yet implemented in code. (2026-06-28: the "Switch to ANSWER/ACTIVE MODE" buttons in the Tag Inspector's Pro Diagnostics panel are now disabled in the UI so the fabricated command can't be sent accidentally before this fix lands — see Resolved below.)

2. **Read TID (Bank 2) is likely blocked by #1, not actually broken itself.** The command `7CFFFF123203020006` (`index.html`) is protocol-correct — CID1=12H, CID2=32H, MB=02 (TID), SA=00, DL=06 words, matches `PROTOCOL_SPEC.md` Section 4.7 exactly. Theory: since mode-switch never worked, the reader is still in factory Active mode, constantly auto-broadcasting reads, and on-demand commands collide with that stream. The `0x1F` "error" referenced in earlier notes does **not** appear anywhere in the manual (only `00`=Success, `01`=Fail, `32`=Auto-send-to-SU are real RTN codes per Table 2-3) — unlike #1, this one is genuinely unverified, not disproven; the manual not mentioning it doesn't mean this clone reader doesn't really emit it. Tracked as "Set Aside" item 2 in `PROTOCOL_SPEC.md` Section 6. Fix #1 first; this may resolve itself. (2026-06-28: the "Read TID (Bank 2)" button is now disabled in the UI alongside the mode-switch buttons — see #1 and Resolved below.)

3. ~~`MockBleDriver` frame shape disagrees with the real `BleDriver`.~~ **Resolved 2026-06-30** — see Resolved section below.

4. **Ingestion-side dedup is documented but not implemented.** `PROJECT_GUIDELINES.md` §4 calls for an "Ingestion Dedup" (don't save the same chip 100x/sec) ahead of the Peak-RSSI export-time dedup. `TimingEngine.handleIncomingTag()` pushes every single read straight to the write queue with no filtering at all. Only the export-time Peak-RSSI 10s-window logic exists (and that one is correct and tested). In real Active-mode scanning against an antenna this could make `race_reads` grow very large during a live race. Not urgent for a one-shot test but will matter at race scale.

5. **Read vs. Write `SA` pointer units disagree within the manual itself.** Section 4.6 says Write's `SA` is a byte pointer; Section 4.7 says Read's `SA` is a word pointer — both use the same example value. This directly affects whether `writeBibToTag()`'s assumption about EPC word offset 2 is correct (see Unconfirmed list below). Documented in `PROTOCOL_SPEC.md` Section 5.

6. ~~`establishConnection()` reports "READER ONLINE" success even if zero characteristics were ever found.~~ **Resolved 2026-06-30** — see Resolved section below.

7. **`parseFrame()` has no cross-notification reassembly buffer.** `BleDriver.js` parses each `characteristicvaluechanged` event's bytes in isolation with zero carry-over. If a single protocol frame's total length ever exceeds the negotiated BLE notification payload size and arrives split across two or more separate events, the first partial chunk fails its `frameEnd > bytes.length` check and is silently dropped — no warning, nothing. Not currently confirmed to be biting anything (a real 41-byte Get Version reply decoded fine, implying this hardware/Chrome/Windows combo negotiates a large-enough MTU in practice), but a longer response (e.g. multi-tag identify with several EPCs) could hit it. Would need a deliberately oversized real response to confirm the actual risk either way.

8. **Chip-to-Bib mapping has zero duplicate protection — two distinct gaps.** `TimingEngine.saveMapping()` (`TimingEngine.js`) does a raw IndexedDB `put()` keyed on `chip_hex`, with no checks at all:
   - **Re-mapping an already-mapped chip** silently overwrites the old bib assignment — no warning that a mapping is being replaced, no record of what it used to be. Desired behavior (per Tony, 2026-06-28): warn + confirm, e.g. *"Chip already mapped to Bib 104 — remapping to Bib 207 will remove that mapping. Continue?"*, showing the mapping that's about to be lost.
   - **Same bib assigned to two different chips** is entirely unguarded, since the store's keyPath is `chip_hex` not `bib_num` — nothing stops bib 104 from being saved against two different chip hexes. This is the more dangerous gap: it would silently produce two separate finish times both labeled bib 104 in the CSV export (`app.exportCsv`, `index.html`), with no indication anything's wrong.
   - Both gaps are also reachable via the bulk CSV import path (`app.importChipMap`, `index.html`), which loops over `saveMapping()` with no validation either.
   - Deliberately not started yet: the right fix depends on what the `chip_map` store's key/lookup strategy should actually be (chip_hex-keyed with a secondary bib_num uniqueness check? something else?), and that needs real thought plus tests before touching `saveMapping()` — not a same-session opportunistic fix. Scope when picked up: warn + confirm for both scenarios (not a hard block), per Tony's direction.

## Resolved

- **Tag Inspector now decodes tag-read frames into separate, labeled fields instead of just the generic header/addr/cid1/cid2/length/info/checksum grid.** `BleDriver.processValidFrame()`'s tag-decode logic (EPC_LEN, EPC bytes, RSSI raw/dBm) was extracted into a standalone `decodeTagFrame(frame)` method, reused by both the real `onTagRead` path and the new "TAG READ DECODE" panel in the Tag Inspector tab — one source of truth, so the inspector can never silently disagree with what real race scoring does. `onRawFrame` now passes `{hex, frame, checksumValid, tagDecode}` instead of a bare hex string, so the inspector can also show a ✓/✗ checksum-valid indicator it never could before (`processValidFrame` previously fired `onRawFrame` *before* even computing the checksum). Covered by 4 new tests in `tests/regression.test.js` (31/31 passing). Found and fixed in the same pass: `MockBleDriver.playSequence()` was still emitting the old bare-hex-string `onRawFrame` shape, which would have crashed the inspector in Mock Mode — updated to match the new contract (`tagDecode: null`, since mock frames don't share the real CID1=0x20 push-frame shape — see #3 below). Also found and fixed: `runBitSearch()` (`index.html`) was reading the EPC via a fragile `document.querySelector('#inspectorDetail div div:nth-child(2) span')` that actually grabbed the **Header** field's span, not the EPC — the Bit-Search Decoder had likely never searched real EPC bytes. Now reads `#inspectorEpcHex`, a stable ID on the new EPC field. Verified by rendering `AppUI.updateInspector()` against a real DOM (happy-dom) for a valid tag frame, a non-tag frame, and a bad-checksum tag frame, and inspecting the actual HTML output — no in-browser check was done (no browser-automation tool available this session).
- **EPC decode fixed and confirmed against 5 real scans (2026-06-28).** The previous "EPC_LEN byte at `frame[6]`" assumption was empirically falsified: Tony provided 4 real race tag-read frames (Bib 21251 & 5251 from one race/provider, Bib 982 & 868 from two other races/providers) plus the manufacturer's own sample/test tag bundled with the reader, and on every single one `frame[6]` was `0x00` — meaning the old code decoded an **empty-string EPC for every real tag read**, with zero ability to distinguish bibs from each other on actual hardware. All 5 checksums were hand-verified (sum of all bytes mod 256 = 0) before trusting the data. The real layout is `INFO = [AN(1B antenna)][PC(2B, standard EPC Gen2 Protocol Control word)][EPC(PC-word-derived length)][RSSI(1B)]` — EPC length is the top 5 bits of PC's high byte, in 16-bit-word units, exactly like a real Gen2 PC word. This was confirmed multiple ways: it correctly predicted a 12-byte EPC for the Race-1 pair (`PC=0x3400`) and the sample tag (`PC=0x3000`), and a 16-byte EPC for the two other-race tags (`PC=0x4000`), every time exactly accounting for the remaining `LENGTH` bytes with nothing left over — and the sample tag's decoded EPC happens to lead with `E2`, the standard Gen2 General Identifier header, which a wrong byte offset wouldn't coincidentally produce. `BleDriver.decodeTagFrame()` now implements this layout; `AppUI.updateInspector()`'s "TAG READ DECODE" panel now shows AN/PC/EPC-length/EPC/RSSI as separate fields. All 5 real frames are now permanent regression test data in `tests/regression.test.js` (36/36 passing), alongside two corrected hand-built synthetic frames (used by the concatenated-frames and bit-flip tests) that now follow the same confirmed shape instead of the old falsified one.
- **Stale regression test rewritten** (`tests/regression.test.js`, "concatenated tag frames" test). The old version used a fixed 7-byte mock frame with no real LENGTH field — it modeled a format that was never real, predating both the protocol read-through and the CID1-based dispatch logic. Rewritten with protocol-accurate frames (real LENGTH-based framing, verified checksum); updated again 2026-06-28 to use the confirmed AN+PC+EPC+RSSI shape instead of the now-falsified EPC_LEN-byte shape.
- **"Bit Flip Test" fixed to fail for the correct reason.** The old version's mock frame got rejected by the *framing* check before checksum validation was ever reached, so it passed without exercising checksum-mismatch logic at all. Rebuilt with correct framing and a deliberately bit-flipped CHKSUM byte, plus a `console.warn` spy proving the rejection now genuinely happens in the checksum branch.
- **"Connect Reader" flakiness investigated and mitigated.** Symptom: `Discovery Error: Connection Error: Connection attempt failed.` on every attempt. Two distinct failure modes share that symptom:
  1. *Ordinary transient flakiness* — `device.gatt.connect()` is known-flaky on Windows right after a BLE scan. Fixed: `BleDriver.js`'s `connectGatt(maxAttempts=4)` retries with backoff and an explicit `gatt.disconnect()` between attempts before giving up.
  2. *Wedged OS Bluetooth radio* — confirmed via `chrome://bluetooth-internals/#devices` showing the device as "Not Connected" (ruling out a stuck Chrome GATT session or another tab holding it). No code can fix this — Web Bluetooth has no permission to reset the OS radio. The only real fix was toggling Bluetooth off/on in Windows Settings.
  - Also fixed: `connect()` and `establishConnection()` used to both call `updateStatus()` on the same bubbled-up error, producing a confusing nested `"Discovery Error: Connection Error: ..."` message. Now cleanly separated — picker-level failures say "Discovery Error", GATT-level failures say "Connection Error", each exactly once.
  - **Runbook for next time this recurs:** check `chrome://bluetooth-internals/#devices` first. "Connected" → something (another tab, a stale handle) is holding it, force-disconnect there. "Not Connected" but still failing after retries → it's the OS radio, toggle Bluetooth off/on in Windows Settings and/or power-cycle the reader.

- **Connection hardening + persistent pairing (2026-06-30).** Five things landed together: (1) `establishConnection()` now throws "No usable characteristics found" instead of silently reporting READER ONLINE when no `ffe` service is present — the old characterization test was flipped to `rejects` first per §7. (2) `requestDevice()` now uses `filters: [{services: ['ffe0']}]` instead of `acceptAllDevices: true`, so the device picker only shows readers that actually advertise the right service. (3) `connect()` saves `device.id` / `device.name` to `localStorage` after a successful pairing. (4) `disconnect()` always clears those keys — intentional disconnect = forget the device, so auto-reconnect can't silently re-pair to a "forgotten" reader. (5) New `tryAutoConnect()` method uses `navigator.bluetooth.getDevices()` (Chrome 85+) to reconnect without showing the picker if the previously paired device is available; `AppUI.restoreState()` calls it on page load. UI: "Connect Reader" button becomes "Disconnect" when connected (label managed by `updateBleBadge`); a "Forget Device" button appears when disconnected with a saved device, to manually clear the saved pairing without having to connect first. Covered by 11 new tests in `connection.test.js` and 5 new tests in `inspector-ui.test.js`.

- **`MockBleDriver` frame shape fixed (2026-06-30).** `MockBleDriver.js` now builds protocol-accurate `CID1=0x20` push frames (`AN+PC+EPC+RSSI` layout) via `buildTagFrame(epcHex, rssiDbm)`, using the same PC-word encoding that real frames use. `playSequence()` and `startSimulation()` both call `buildTagFrame()` → `decodeTagFrame()` (an inline mirror of `BleDriver.decodeTagFrame()`) and emit the result as a proper `tagDecode` object in `onRawFrame` — so Mock Mode now exercises the same parse path as real hardware, and the Tag Inspector shows live decode output in Mock Mode. The old `tag`-as-raw-frame-hex sequence format has been replaced with `{ epcHex, rssi, delay }`. Covered by `tests/mock-driver.test.js` (19 tests), including a cross-validation test that independently runs `BleDriver.decodeTagFrame()` against the frames `MockBleDriver` emits to confirm they agree.

- **Tag Inspector's WIP/dangerous controls are now disabled in the UI (2026-06-28), not just documented as risky.** The Pro Tag Writer ("Burn to Tag"), Pro Diagnostics (mode-switch, Get Version, Read TID), Manual Command (Advanced) (both send buttons), and Bit-Search Decoder controls are all `disabled` in `index.html` — kept visible with a "(WIP — disabled, see KNOWN_ISSUES.md)" label rather than removed, so the half-finished/unverified features (#1, #2, and the two Unconfirmed items above) can't be triggered by accident (e.g. mid-race) before they're actually fixed. The passive Tag Inspector display (Last Scanned Frame, Frame History) is untouched — only the active/write/command controls are gated. Covered by a new `tests/inspector-ui.test.js` (5 tests, loads the real `index.html` body into happy-dom and asserts each control's `disabled` state), so a future edit can't silently re-enable one of these without a test failing.

## Unconfirmed / Low Priority

- `BleDriver.processValidFrame`'s RSSI sign handling (`rssiRaw > 127 ? rssiRaw - 256 : -rssiRaw`) looks like a reverse-engineered heuristic rather than something pulled from a documented RSSI field — the manual never mentions RSSI at all. Worth a second look if RSSI-based decisions (e.g. Peak-RSSI export logic) ever look suspect.
- `AppUI.writeBibToTag()` assumes the bib number is written as a raw 16-bit value directly into the EPC bank at word offset 2 (skipping the PC word) — unverified against spec or a real written/read-back tag. Complicated by #5 above: if `SA` for EPC writes is actually a byte pointer rather than a word pointer, this write may be landing 2 bytes off from where it's assumed to be. Disabled in the UI for now (see Resolved below). Related idea worth real design time later (per Tony, 2026-06-28): the 5-real-scan EPC analysis confirmed bib numbers are *not* encoded in any vendor-supplied chip's EPC (checked hex/decimal/BCD/ASCII substrings and byte-window matches — none hit), so if blank chips ever get bought intentionally (cheaper than pre-encoded ones), self-encoding the bib number via a fixed `writeBibToTag()` would be the way to make that work — but that's blocked on resolving the SA-pointer ambiguity first, not a same-session fix.
- The "Bit-Search Decoder" feature (`app.runBitSearch`) is a manual debug heuristic (search for a bib's binary pattern inside the raw EPC) — fine as a debug tool, just not a real decode of a structured tag ID. Disabled in the UI for now (see Resolved below).

## How to Use This Doc

Read this alongside `PROTOCOL_SPEC.md` and `PROJECT_GUIDELINES.md` before making changes here, regardless of which machine you're on. Pull from this list rather than re-deriving findings from scratch. Update it (move items to Resolved, add new findings) as work progresses — don't let it go stale. Per `PROJECT_GUIDELINES.md` §5 and §7: no code change without the User's explicit go-ahead, and no code change without a failing test first.
