# Known Issues

Running punch-list of confirmed bugs and open questions in OrcStompTiming, to tackle deliberately rather than all at once. This file lives in git on purpose â€” it's the source of truth across machines and sessions, not anyone's local notes.

Context: this is a DIY RFID 5k timing rig built around a cheap Chinese UHF reader (manufacturer `aosid.com`, type code `ABQ`, firmware `V1.22`), built as a proof of concept to avoid a $5k commercial solution. The codebase has rough edges by design â€” this list exists so fixes get applied one at a time, deliberately, instead of guessed at from scratch each session. See [PROTOCOL_SPEC.md](PROTOCOL_SPEC.md) for the reader's command protocol and [PROJECT_GUIDELINES.md](PROJECT_GUIDELINES.md) for the rules governing how changes get made here.

## Open Issues

1. ~~Mode-switch command was fabricated, not real protocol.~~ **Resolved 2026-07-05** â€” see Resolved section below.

2. **Read TID (Bank 2) is likely blocked by Active-mode collisions, not actually broken itself.** The command `7CFFFF123203020006` (`index.html`) is protocol-correct â€” CID1=12H, CID2=32H, MB=02 (TID), SA=00, DL=06 words, matches `PROTOCOL_SPEC.md` Section 4.7 exactly. Theory: in Active mode the reader auto-broadcasts reads and on-demand commands collide with that stream. Now that `setWorkMode()` is implemented (#1 resolved), the real test is: switch to Command mode via "Switch to ANSWER MODE" â†’ click "Read TID (Bank 2)" â†’ does it respond? That experiment hasn't been run yet. The `0x1F` "error" referenced in earlier notes does **not** appear anywhere in the manual (only `00`=Success, `01`=Fail, `32`=Auto-send-to-SU are real RTN codes per Table 2-3) â€” unlike #1, this one is genuinely unverified, not disproven; the manual not mentioning it doesn't mean this clone reader doesn't really emit it. Tracked as "Set Aside" item 2 in `PROTOCOL_SPEC.md` Section 6. The "Read TID (Bank 2)" button remains disabled in the UI until this is verified.

3. ~~`MockBleDriver` frame shape disagrees with the real `BleDriver`.~~ **Resolved 2026-06-30** â€” see Resolved section below.

4. ~~Ingestion-side dedup is documented but not implemented.~~ **Resolved 2026-07-01** â€” see Resolved section below.

5. **Read vs. Write `SA` pointer units disagree within the manual itself.** Section 4.6 says Write's `SA` is a byte pointer; Section 4.7 says Read's `SA` is a word pointer â€” both use the same example value. This directly affects whether `writeBibToTag()`'s assumption about EPC word offset 2 is correct (see Unconfirmed list below). Documented in `PROTOCOL_SPEC.md` Section 5.

6. ~~`establishConnection()` reports "READER ONLINE" success even if zero characteristics were ever found.~~ **Resolved 2026-06-30** â€” see Resolved section below.

7. **`parseFrame()` has no cross-notification reassembly buffer.** `BleDriver.js` parses each `characteristicvaluechanged` event's bytes in isolation with zero carry-over. If a single protocol frame's total length ever exceeds the negotiated BLE notification payload size and arrives split across two or more separate events, the first partial chunk fails its `frameEnd > bytes.length` check and is silently dropped â€” no warning, nothing. Not currently confirmed to be biting anything (a real 41-byte Get Version reply decoded fine, implying this hardware/Chrome/Windows combo negotiates a large-enough MTU in practice), but a longer response (e.g. multi-tag identify with several EPCs) could hit it. Would need a deliberately oversized real response to confirm the actual risk either way.

8. ~~Chip-to-Bib mapping has zero duplicate protection â€” two distinct gaps.~~ **Resolved 2026-07-05** for the kiosk form path â€” see Resolved below.
9. ~~Bulk CSV chip-map import bypasses the reassignment guardrails added for the kiosk form.~~ **Resolved 2026-07-05** â€” see Resolved below.

## Resolved

- **Tag Inspector now decodes tag-read frames into separate, labeled fields instead of just the generic header/addr/cid1/cid2/length/info/checksum grid.** `BleDriver.processValidFrame()`'s tag-decode logic (EPC_LEN, EPC bytes, RSSI raw/dBm) was extracted into a standalone `decodeTagFrame(frame)` method, reused by both the real `onTagRead` path and the new "TAG READ DECODE" panel in the Tag Inspector tab â€” one source of truth, so the inspector can never silently disagree with what real race scoring does. `onRawFrame` now passes `{hex, frame, checksumValid, tagDecode}` instead of a bare hex string, so the inspector can also show a âś“/âś— checksum-valid indicator it never could before (`processValidFrame` previously fired `onRawFrame` *before* even computing the checksum). Covered by 4 new tests in `tests/regression.test.js` (31/31 passing). Found and fixed in the same pass: `MockBleDriver.playSequence()` was still emitting the old bare-hex-string `onRawFrame` shape, which would have crashed the inspector in Mock Mode â€” updated to match the new contract (`tagDecode: null`, since mock frames don't share the real CID1=0x20 push-frame shape â€” see #3 below). Also found and fixed: `runBitSearch()` (`index.html`) was reading the EPC via a fragile `document.querySelector('#inspectorDetail div div:nth-child(2) span')` that actually grabbed the **Header** field's span, not the EPC â€” the Bit-Search Decoder had likely never searched real EPC bytes. Now reads `#inspectorEpcHex`, a stable ID on the new EPC field. Verified by rendering `AppUI.updateInspector()` against a real DOM (happy-dom) for a valid tag frame, a non-tag frame, and a bad-checksum tag frame, and inspecting the actual HTML output â€” no in-browser check was done (no browser-automation tool available this session).
- **EPC decode fixed and confirmed against 5 real scans (2026-06-28).** The previous "EPC_LEN byte at `frame[6]`" assumption was empirically falsified: Tony provided 4 real race tag-read frames (Bib 21251 & 5251 from one race/provider, Bib 982 & 868 from two other races/providers) plus the manufacturer's own sample/test tag bundled with the reader, and on every single one `frame[6]` was `0x00` â€” meaning the old code decoded an **empty-string EPC for every real tag read**, with zero ability to distinguish bibs from each other on actual hardware. All 5 checksums were hand-verified (sum of all bytes mod 256 = 0) before trusting the data. The real layout is `INFO = [AN(1B antenna)][PC(2B, standard EPC Gen2 Protocol Control word)][EPC(PC-word-derived length)][RSSI(1B)]` â€” EPC length is the top 5 bits of PC's high byte, in 16-bit-word units, exactly like a real Gen2 PC word. This was confirmed multiple ways: it correctly predicted a 12-byte EPC for the Race-1 pair (`PC=0x3400`) and the sample tag (`PC=0x3000`), and a 16-byte EPC for the two other-race tags (`PC=0x4000`), every time exactly accounting for the remaining `LENGTH` bytes with nothing left over â€” and the sample tag's decoded EPC happens to lead with `E2`, the standard Gen2 General Identifier header, which a wrong byte offset wouldn't coincidentally produce. `BleDriver.decodeTagFrame()` now implements this layout; `AppUI.updateInspector()`'s "TAG READ DECODE" panel now shows AN/PC/EPC-length/EPC/RSSI as separate fields. All 5 real frames are now permanent regression test data in `tests/regression.test.js` (36/36 passing), alongside two corrected hand-built synthetic frames (used by the concatenated-frames and bit-flip tests) that now follow the same confirmed shape instead of the old falsified one.
- **Stale regression test rewritten** (`tests/regression.test.js`, "concatenated tag frames" test). The old version used a fixed 7-byte mock frame with no real LENGTH field â€” it modeled a format that was never real, predating both the protocol read-through and the CID1-based dispatch logic. Rewritten with protocol-accurate frames (real LENGTH-based framing, verified checksum); updated again 2026-06-28 to use the confirmed AN+PC+EPC+RSSI shape instead of the now-falsified EPC_LEN-byte shape.
- **"Bit Flip Test" fixed to fail for the correct reason.** The old version's mock frame got rejected by the *framing* check before checksum validation was ever reached, so it passed without exercising checksum-mismatch logic at all. Rebuilt with correct framing and a deliberately bit-flipped CHKSUM byte, plus a `console.warn` spy proving the rejection now genuinely happens in the checksum branch.
- **"Connect Reader" flakiness investigated and mitigated.** Symptom: `Discovery Error: Connection Error: Connection attempt failed.` on every attempt. Two distinct failure modes share that symptom:
  1. *Ordinary transient flakiness* â€” `device.gatt.connect()` is known-flaky on Windows right after a BLE scan. Fixed: `BleDriver.js`'s `connectGatt(maxAttempts=4)` retries with backoff and an explicit `gatt.disconnect()` between attempts before giving up.
  2. *Wedged OS Bluetooth radio* â€” confirmed via `chrome://bluetooth-internals/#devices` showing the device as "Not Connected" (ruling out a stuck Chrome GATT session or another tab holding it). No code can fix this â€” Web Bluetooth has no permission to reset the OS radio. The only real fix was toggling Bluetooth off/on in Windows Settings.
  - Also fixed: `connect()` and `establishConnection()` used to both call `updateStatus()` on the same bubbled-up error, producing a confusing nested `"Discovery Error: Connection Error: ..."` message. Now cleanly separated â€” picker-level failures say "Discovery Error", GATT-level failures say "Connection Error", each exactly once.
  - **Runbook for next time this recurs:** check `chrome://bluetooth-internals/#devices` first. "Connected" â†’ something (another tab, a stale handle) is holding it, force-disconnect there. "Not Connected" but still failing after retries â†’ it's the OS radio, toggle Bluetooth off/on in Windows Settings and/or power-cycle the reader.

- **Connection hardening + persistent pairing (2026-06-30).** Five things landed together: (1) `establishConnection()` now throws "No usable characteristics found" instead of silently reporting READER ONLINE when no `ffe` service is present â€” the old characterization test was flipped to `rejects` first per Â§7. (2) `requestDevice()` uses `acceptAllDevices: true` with `optionalServices: ['ffe0', 'ffe1']` â€” a `filters: [{services: ['ffe0']}]` approach was attempted but reverted (2026-06-30) because this hardware does not include `ffe0` in its BLE advertisement packets, causing the device picker to show nothing. (3) `connect()` saves `device.id` / `device.name` to `localStorage` after a successful pairing. (4) `disconnect()` always clears those keys â€” intentional disconnect = forget the device, so auto-reconnect can't silently re-pair to a "forgotten" reader. (5) New `tryAutoConnect()` method uses `navigator.bluetooth.getDevices()` (Chrome 85+) to reconnect without showing the picker if the previously paired device is available; `AppUI.restoreState()` calls it on page load. **Confirmed non-functional on Windows Chrome (2026-06-30):** `getDevices()` consistently returns an empty list even immediately after a successful `requestDevice()` grant on `localhost` â€” the code is correct but the platform doesn't honour it. No JS workaround exists; see WISHLIST Auto-Connect entry. UI: "Connect Reader" button becomes "Disconnect" when connected (label managed by `updateBleBadge`); a "Forget Device" button appears when disconnected with a saved device, to manually clear the saved pairing without having to connect first. Covered by 11 new tests in `connection.test.js` and 5 new tests in `inspector-ui.test.js`.

- **L2 ingestion dedup implemented (2026-07-01).** `TimingEngine` now maintains a `seenTags: Map<tagHex, firstSeenMs>` that gates writes in `startDaemon()`. The first read for a tag always writes; subsequent reads within the 10-second window also write (so L3 peak-RSSI selection still works correctly across the full window); reads arriving after the window closes are silently dropped at the daemon, not written to IndexedDB at all. `clearRaceData()` clears the map so each new race starts fresh. This resolves the DB-growth concern without touching L3 (export-time peak selection is unchanged and still correct). Covered by 7 tests in `tests/dedup.test.js`.

- **`MockBleDriver` frame shape fixed (2026-06-30).** `MockBleDriver.js` now builds protocol-accurate `CID1=0x20` push frames (`AN+PC+EPC+RSSI` layout) via `buildTagFrame(epcHex, rssiDbm)`, using the same PC-word encoding that real frames use. `playSequence()` and `startSimulation()` both call `buildTagFrame()` â†’ `decodeTagFrame()` (an inline mirror of `BleDriver.decodeTagFrame()`) and emit the result as a proper `tagDecode` object in `onRawFrame` â€” so Mock Mode now exercises the same parse path as real hardware, and the Tag Inspector shows live decode output in Mock Mode. The old `tag`-as-raw-frame-hex sequence format has been replaced with `{ epcHex, rssi, delay }`. Covered by `tests/mock-driver.test.js` (19 tests), including a cross-validation test that independently runs `BleDriver.decodeTagFrame()` against the frames `MockBleDriver` emits to confirm they agree.

- **Tag Inspector's WIP/dangerous controls are now disabled in the UI (2026-06-28), not just documented as risky.** The Pro Tag Writer ("Burn to Tag"), Pro Diagnostics (mode-switch, Get Version, Read TID), Manual Command (Advanced) (both send buttons), and Bit-Search Decoder controls are all `disabled` in `index.html` â€” kept visible with a "(WIP â€” disabled, see KNOWN_ISSUES.md)" label rather than removed, so the half-finished/unverified features (#1, #2, and the two Unconfirmed items above) can't be triggered by accident (e.g. mid-race) before they're actually fixed. The passive Tag Inspector display (Last Scanned Frame, Frame History) is untouched â€” only the active/write/command controls are gated. Covered by a new `tests/inspector-ui.test.js` (5 tests, loads the real `index.html` body into happy-dom and asserts each control's `disabled` state), so a future edit can't silently re-enable one of these without a test failing.

- **Chip-to-Bib reassignment guardrails added for the kiosk form (2026-07-05).** Closes both gaps in old item #8 for manual/kiosk entry. `TimingEngine.saveMapping()` now coerces `bib_num` via `parseInt()` regardless of caller type (fixes a related bug found this session: the kiosk form passed a raw `<input>.value` string while CSV import already `parseInt`'d, so the same bib could be stored as `"104"` and `104` and never compare equal). New `AppUI.checkMappingConflicts(chipHex, bibNum, maps)` detects both a chip-side conflict (this chip already mapped to a different bib) and a bib-side conflict (this bib already held by a different chip); `AppUI.submitMapping()` shows a `confirm()` naming the conflicting side(s) before saving, and does nothing silently â€” cancelling leaves the mapping untouched. `AppUI.refreshMappingFormState()` reflects this live as the operator types/scans: a high-vis "ALREADY MAPPED" banner (`#mappingConflictWarning`) and the submit button relabeling to "REASSIGN" (`#mappingSubmitBtn`), wired via `oninput` on both fields and called at the end of `fillKioskForm()` so a kiosk scan of an already-registered chip shows the warning immediately, before the operator types anything. Also fixed while in the area: `renderMappingTable()` now escapes `chip_hex`/`bib_num` before interpolating into `innerHTML` (a crafted CSV import row could previously inject markup into the registry table). Caught via manual browser smoke-testing (Playwright + headless Chromium), not just unit tests: the first implementation warned+confirmed a bib-side steal but only wrote the new chip's row, leaving both chips sharing the bib â€” `chip_map` is keyed by `chip_hex`, so `saveMapping()` had no way to touch the losing chip's row. Fixed by adding `TimingEngine.deleteMapping(chipHex)` and having `submitMapping()` call it on the losing chip once a bib steal is confirmed; `index.html`'s `app.deleteMapping` now also delegates to it instead of touching IndexedDB directly. Covered by `tests/chip-map.test.js` (38 tests).

- **Bulk CSV chip-map import rebuilt as an all-or-nothing full replace (2026-07-05), per Tony's direction.** Old `app.importChipMap` looped calling `saveMapping()` per row with zero validation â€” same duplicate-bib/silent-overwrite risk as #8, but for an entire file at once. New `AppUI.parseChipMapCsv(text)` parses rows (normalized like `saveMapping`, tagged with the 1-indexed source line number) and `AppUI.findCsvConflicts(rows)` checks the file's own internal consistency *before* it ever touches `chip_map`: a chip_hex repeated across rows, or a bib_num claimed by two different chips, both refuse the import via `alert()` naming the exact conflicting chip/bib and line numbers â€” no partial import, existing registry left untouched. A file with zero valid rows is also refused (guards against silently wiping the registry on a garbage/empty upload). If the file is clean, `AppUI.submitChipMapImport()` shows a `confirm()` making clear this **replaces the entire registry** (not a merge â€” anything not in the file is deleted), and only calls new `TimingEngine.replaceChipMap(mappings)` (clears `chip_map`, inserts the file's rows in one transaction) if accepted. `index.html`'s `app.importChipMap` is now a thin wrapper. Verified in a real browser (Playwright + headless Chromium): conflicting file refused with correct line numbers and registry untouched, clean file's overwrite warning shown, cancel preserved the existing registry, accept replaced it â€” zero console errors. Covered by `tests/chip-map.test.js` (54 tests total in that file).
